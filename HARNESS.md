# HARNESS.md — 작업 하네스 설계 기준

우석의 모든 작업 요청에 이 구조를 기본으로 적용한다.
Anthropic의 3-agent 하네스 설계에서 영감을 받아 claw 방식으로 정제함.

---

## 핵심 구조: Planner → Generator → Evaluator

### 1. Planner (계획 단계)
작업을 받으면 바로 실행하지 않는다. 먼저 다음을 확인한다:
- 작업을 어떤 단위로 쪼갤 것인가?
- 각 단위는 독립적으로 실행 가능한가?
- 최종 결과물의 기준(Evaluation Criteria)은 무엇인가?

복잡한 작업(보고서, 브리핑, 영상 스크립트, 발표 정리 등)은 **계획 먼저 보여주고 승인 받은 뒤** 실행한다.
단순한 작업(캘린더 등록, 빠른 검색 등)은 바로 실행해도 된다.

### 2. Generator (실행 단계)
- 서브에이전트로 각 파트 독립 실행
- 컨텍스트 오염을 막기 위해 서브에이전트는 필요한 정보만 받는다
- 하나의 서브에이전트가 너무 많은 걸 하지 않도록 쪼갠다

### 3. Evaluator (평가 단계)
생성 후 별도 에이전트(또는 claw 자신)가 냉정하게 평가한다.
절대 "잘 됐습니다"로 끝내지 않는다. 아래 기준으로 채점한다.

---

## 결과물 평가 기준 (우석 맞춤)

### 보고서 / 브리핑 / HTML 결과물
| 기준 | 설명 | 가중치 |
|---|---|---|
| **디자인 품질** | 색상, 타이포, 레이아웃이 일관된 하나의 분위기를 만드는가? | 높음 |
| **독창성** | AI 템플릿 냄새가 나지 않는가? 의도적인 디자인 선택이 보이는가? | 높음 |
| **완성도** | 타이포 위계, 여백, 색상 대비, 모바일 대응이 올바른가? | 중간 |
| **기능성** | 읽기 쉽고, 핵심 정보를 빠르게 파악할 수 있는가? | 중간 |

### 콘텐츠 / 스크립트 / 요약 결과물
| 기준 | 설명 |
|---|---|
| **명확성** | 우석의 수준에 맞게 쉽고 명확하게 설명되어 있는가? |
| **인사이트** | 단순 정보 나열이 아니라 "왜 중요한지"가 담겨 있는가? |
| **액션 연결** | 우석이 실제로 취할 수 있는 행동으로 이어지는가? |
| **간결함** | 불필요한 내용 없이 핵심만 담겨 있는가? |

## 실제 적용 방식

### 복잡한 작업 요청 시 (보고서, 브리핑, 발표 정리 등)
```
1. claw → 계획 제시 (파트 분리 + 평가 기준 명시)
2. 우석 → 승인 or 수정
3. 서브에이전트들 → 각 파트 독립 실행
4. Evaluator 에이전트 → 결과물 채점 + 개선 포인트 리스트
5. 필요 시 Generator 재실행
6. 최종 결과물 → 우석에게 전달 + 채점 결과 요약
```

### 단순 작업 (캘린더, 빠른 검색, 리마인더 등)
```
바로 실행. 하네스 불필요.
```

## 언제 하네스를 쓰는가?

**적용:**
- 보고서/브리핑 제작
- 더바코 관련 콘텐츠 (숏츠 스크립트, 마케팅 자료)
- 포르쉐/LR 발표 정리
- 웹페이지/HTML 제작
- 리서치 + 분석 작업
- 복잡한 의사결정 지원

**미적용:**
- 캘린더 등록
- 단순 검색/요약
- 리마인더 설정
- 빠른 질문/답변

---

## Self-Evaluation 원칙

claw는 자기가 만든 결과물을 칭찬하지 않는다.
평가 시 항상 "뭐가 아쉬운지"를 먼저 찾는다.
"좋은 것 같아요" 대신 "이 부분이 약하고, 이렇게 개선할 수 있어요"로 끝낸다.
