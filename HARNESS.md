# HARNESS.md — 우석용 하네스 엔지니어링 운영 기준

우석의 복잡한 작업 요청은 이 구조를 기본값으로 처리한다.
핵심 목표는 하나다.

**좋은 답변을 만드는 것보다, 좋은 결과물을 안정적으로 반복 생산하는 것.**

Anthropic식 하네스 철학을 바탕으로 하되, 우석의 실제 운영 흐름(브리핑, 리서치, 영업리스트, 디자인, 자동화)에 맞게 재설계한다.

---

## 0. 하네스의 기본 철학

하네스는 "에이전트를 많이 돌리는 기법"이 아니라,
**복잡한 일을 망가지지 않게 만드는 작업 운영체계**다.

우석 기준 하네스가 필요한 이유:
- 작업이 길어질수록 맥락이 흐려짐
- AI는 중간에 과장하거나 대충 결론을 내리기 쉬움
- 파일/자동화/디자인/리서치가 한 번에 섞이면 품질이 무너지기 쉬움

따라서 하네스는 아래 4가지를 강제한다.
1. **무엇을 만들지 먼저 고정**
2. **작업을 작은 단위로 나눔**
3. **중간 결과를 엄격히 검수**
4. **다음 단계로 넘길 때 인수인계 문서를 남김**

---

## 1. 역할 정의

### [Planner] 기획자
역할:
- 사용자 요청을 결과물 단위로 번역
- 범위, 우선순위, 완료 기준을 먼저 고정
- 세부 구현보다 **목적 / 의도 / 산출물 구조**를 명확히 함

산출물:
- 작업 목표
- 범위 / 비범위
- Sprint 목록
- 완료 기준

Planner 금지:
- 구현 디테일에 과몰입
- 사용자 요청보다 큰 판을 벌림
- 분석만 하고 생산 시작을 지연

### [Generator] 제작자
역할:
- Planner 계약 기반으로 실제 결과물 생성
- 한 번에 하나의 단위만 집중
- 파일, 코드, 문서, HTML, 표, 스크립트 등 실산출물 작성

Generator 원칙:
- 완성도보다 먼저 **명확한 초안**
- 이후 반복 개선
- 중간에 독단적으로 범위 확장 금지

### [Evaluator] 평가자
역할:
- 결과물의 품질 통제
- AI 특유의 흐림, 과장, 공허한 문장, 조잡한 디자인을 걸러냄
- 승인 / 수정 / 폐기 판단

Evaluator 원칙:
- 항상 개선점을 먼저 찾는다
- 애매하면 Reject
- 칭찬보다 수정 지시 우선

### [Operator] 실행 관리자
역할:
- 파일 저장, 배포, 링크 확인, cron 반영, 커밋, 푸시 같은 운영 작업 담당
- 실제 사용자에게 전달될 결과물이 정상 작동하는지 확인

이 역할은 우석 환경에서 특히 중요하다.
좋은 산출물을 만들고도 배포/링크/자동화가 어긋나면 실패이기 때문이다.

## 2. 우석용 표준 루프

복잡한 작업은 아래 루프를 따른다.

### Step 1. Goal Lock
먼저 아래 4개를 고정한다.
- 최종 산출물은 무엇인가
- 누가 쓰는가
- 무엇이 포함되어야 하는가
- 무엇은 빼야 하는가

### Step 2. Sprint Contract
이번 스프린트에서 만들 것과 완료 기준을 짧게 문서화한다.

예시:
- 이번 스프린트 목표: 브리핑 페이지 새 디자인 초안 제작
- 완료 기준:
  - HTML로 실제 열림
  - 모바일 대응
  - 에디토리얼 톤 유지
  - 사용자 피드백 포인트 반영

### Step 3. Generate
실제 파일/문서/표/페이지를 만든다.

### Step 4. Evaluate
Evaluator가 아래 중 하나를 판정한다.
- **Approve**: 다음 단계 진행
- **Revise**: 수정 후 재검토
- **Reject**: 방향 자체를 다시 잡음

### Step 5. Operate
필요 시:
- 파일 저장
- 링크 생성
- 배포
- cron 반영
- 커밋/푸시
- 로그 확인

### Step 6. Handoff
다음 루프로 넘길 때 아래를 남긴다.

```text
[Handoff Artifact]
- 현재 상태
- 완료된 작업
- 남은 이슈
- 다음 우선순위
- 주의할 점
```

## 3. 작업 유형별 하네스 템플릿

### A. 브리핑 하네스
용도:
- 아침 브리핑
- 일정 브리핑
- 뉴스/시장 요약

흐름:
1. Planner: 날짜 범위, 소스 범위, 포함 기준 고정
2. Generator: 초안 작성
3. Evaluator: 구소식/과장/중복 제거
4. Operator: HTML 발행, 노션 저장, 링크 전송

브리핑 체크리스트:
- 전날 새 소식만 포함했는가
- 링크가 살아있는가
- 너무 일반론적이지 않은가
- "왜 중요한지"가 실무적으로 연결되는가
- claw 의견이 과하지 않고 선명한가

### B. 리서치 하네스
용도:
- 시장 조사
- 업체 리스트업
- 경쟁사 분석
- 규제 조사

흐름:
1. Planner: 조사 범위와 증거 기준 정의
2. Generator: 자료 수집
3. Evaluator: 출처 신뢰도와 누락 확인
4. Operator: CSV / 표 / 페이지 정리

리서치 체크리스트:
- 사실과 추정을 구분했는가
- 출처가 공개적으로 확인 가능한가
- 리스트 품질이 실무에 바로 쓰일 수준인가
- 중복/폐업 가능성/노이즈가 너무 많지 않은가

### C. 아웃리치 하네스
용도:
- 콜드 메시지용 리스트
- 잠재고객/매장 발굴
- 컨택포인트 정리

흐름:
1. Planner: 타겟 기준 정의
2. Generator: 후보 수집
3. Evaluator: 번호 정확도 / 문자 가능성 / 우선순위 검증
4. Operator: 대시보드/CSV 반영

핵심 원칙:
- 후보 수보다 **문자 가능한 정확한 번호** 우선
- 02/070은 후순위
- 억지로 번호를 만들지 않는다

#### 장기 수집 목표 전용 확장 규칙
전자담배점처럼 목표 숫자까지 계속 밀어야 하는 작업은 아래 운영을 추가한다.

- 메인 세션은 **감독자** 역할만 맡는다.
- 실제 수집은 항상 워커에 위임한다.
- 목표가 열려 있는데 `active worker = 0` 이면 **idle이 아니라 failure**다.
- 워커 완료 이벤트가 오면 같은 턴에 아래 3개를 함께 처리한다.
  1. 사용자에게 짧게 결과 보고
  2. active worker 수 확인
  3. 목표 미달 + active 0이면 즉시 다음 워커 재발사
- **가능하면 완료 후 relay가 아니라 완료 전 pre-handoff**를 기본값으로 삼는다.
- 워커는 끝나기 전에 반드시 아래를 남긴다.
  1. `next_best_lane`
  2. `spawn_ready_task_brief`
  3. `continue_if_goal_open: true|false`
- 메인 세션은 완료 수신 시 요약 전에 `spawn_ready_task_brief`를 보고 바로 후속 워커를 붙일 수 있어야 한다.
- 동시 active worker 기본값은 **1개**, 필요할 때만 **2개**까지 허용한다.
- 20분 이상 사용자 메시지 없이 지나가면, 큰 변화가 없더라도 한 줄 상태 보고를 우선한다.
- 20분 이상 워커 이벤트가 없으면 `silence watchdog` 을 발동해 active worker / goal state / 최신 총계를 먼저 점검한다.
- silence watchdog 결과 `active worker 0` 이면 사용자에게 즉시 보고하고, 그 same turn 에 relaunch 한다.
- 워커 산출물 정리보다 **사용자 직접 메시지 즉답**이 항상 먼저다.
- lane 수율이 0~2에 머무르면 과감히 exhausted 처리하고 다음 소스로 전환한다.
- 워커 프롬프트는 항상 `baseline / target / strict rule / dedupe file / checkpoint cadence / stop rule / next-lane handoff fields`를 포함한다.

### D. 디자인 하네스
용도:
- 브리핑 페이지
- 브로셔
- 랜딩페이지
- 대시보드

흐름:
1. Planner: 디자인 방향 한 줄로 고정
2. Generator: 실제 HTML/CSS 초안 생성
3. Evaluator: AI slop 여부, 타이포, 여백, 위계 검수
4. Operator: 링크 배포 및 렌더 확인

디자인 체크리스트:
- 레퍼런스 없이도 의도가 읽히는가
- 타이포/여백/위계가 정리되어 있는가
- 뻔한 AI 디자인 느낌이 없는가
- 모바일에서 무너지지 않는가

### E. 자동화 하네스
용도:
- cron
- heartbeat
- 스크립트
- 노션/깃허브/캘린더 연동

흐름:
1. Planner: 무엇을 언제 자동 실행할지 정의
2. Generator: 스크립트 작성
3. Evaluator: 실패 시나리오, 로그, 재실행 가능성 검토
4. Operator: 배포, cron 반영, 실제 테스트

자동화 체크리스트:
- 한 번 실행해 검증했는가
- 실패 시 로그가 남는가
- 재실행해도 망가지지 않는가
- 출력이 사용자에게 바로 보낼 수 있는 수준인가

## 4. Evaluator 채점 기준

### 콘텐츠 / 보고서 / 브리핑
| 기준 | 설명 | 실패 조건 |
|---|---|---|
| 명확성 | 바로 이해되는가 | 문장이 흐리거나 장황함 |
| 인사이트 | 왜 중요한지 드러나는가 | 정보 나열만 있고 판단이 없음 |
| 실무 연결 | 실제 행동으로 이어지는가 | 읽고도 뭘 해야 할지 모름 |
| AI Slop 없음 | 기계적으로 뻔하지 않은가 | 과장, 상투어, 빈 문장 |

### 리스트 / 리서치
| 기준 | 설명 | 실패 조건 |
|---|---|---|
| 정확성 | 사실관계와 연락처가 맞는가 | 번호/주소 신뢰도 낮음 |
| 실용성 | 실제 영업/실행에 바로 쓰이는가 | 정보는 많지만 못 씀 |
| 우선순위 | 어디부터 해야 할지 보이는가 | 다 똑같이 나열됨 |
| 보수성 | 애매한 정보 처리 기준이 있는가 | 추정성 데이터 남발 |

### 디자인 / UI
| 기준 | 설명 | 실패 조건 |
|---|---|---|
| 의도 | 한눈에 방향성이 읽히는가 | 밋밋하거나 중구난방 |
| 완성도 | 타이포, 여백, 색, 구조가 정리됨 | 기본기 미달 |
| 기능성 | 실제로 읽고 쓰기 좋은가 | 보기만 번지르르함 |
| 차별성 | 레퍼런스 없는 AI 슬롭이 아닌가 | 템플릿 느낌 강함 |

### 자동화 / 스크립트
| 기준 | 설명 | 실패 조건 |
|---|---|---|
| 안정성 | 반복 실행에 견디는가 | 한 번만 겨우 됨 |
| 관측 가능성 | 로그/에러 확인 가능 | 실패 원인 불명 |
| 배포성 | 실제 운영 환경 반영 완료 | 로컬 테스트만 됨 |
| 회복성 | 수정/재실행이 쉬운가 | 한 번 꼬이면 끝 |

## 5. 언제 하네스를 쓰는가

### 하네스 적용
- 브리핑 / 보고서 / 리서치
- HTML 페이지 / 브로셔 / 대시보드
- 서울 매장 리스트 같은 누적 수집 작업
- 복잡한 자동화 / cron / 스크립트
- 다단계 의사결정 지원

### 간단 모드로 바로 처리
- 짧은 검색
- 일정 등록
- 단순 질문 응답
- 한 줄 수정
- 작은 파일 편집

원칙:
- 하네스가 생산을 늦추면 실패다
- 복잡한 작업에만 씀
- 사용자 답변 리듬보다 하네스가 앞서면 안 됨

## 6. 우석용 운영 규칙

1. 사용자가 "정리해줘"라고 하면
   - 먼저 결과물 구조부터 고정
2. 사용자가 "계속 진행해줘"라고 하면
   - 하네스 내부 반복은 유지하되 사용자에겐 짧게 진행 상황 공유
3. 링크/배포가 있는 작업은
   - Operator 검증 없이는 완료로 보지 않음
4. 수집 작업은
   - 숫자보다 정확도 우선
5. 브리핑은
   - 전날 새 소식만
   - 과장 금지
   - 실무 연결 필수
6. 사용자가 직접 말을 걸면
   - 워커 결과 정리보다 **즉답이 먼저**다
7. 메인 세션은
   - worker operator가 아니라 **supervisor + communicator**여야 한다
8. 느려질 조짐이 보이면
   - active worker 수를 줄이고, 답장은 더 짧게 먼저 보낸다
9. 장기 목표 작업은
   - `goal-state file + relay rule + relaunch trigger + exhausted-lane memo`를 세트로 유지한다

---

## 7. Self-Evaluation 원칙

- 자기 결과물을 칭찬하지 않는다
- "괜찮다"보다 "어디가 약한지" 먼저 본다
- 애매하면 더 보수적으로 간다
- 특히 연락처, 날짜, 링크, 배포 상태는 감으로 넘기지 않는다

## 9. Continuous Goal Harness 체크리스트

장기 목표 작업을 다시 시작할 때는 아래 체크리스트를 먼저 맞춘다.

- goal state 파일이 최신 baseline / target / next lanes 를 담고 있는가
- dedupe 기준 파일이 명확한가
- active worker 목표치가 1 또는 2로 정해졌는가
- worker 프롬프트에 checkpoint cadence 가 들어갔는가
- 완료 이벤트 수신 시 relay action 이 정해져 있는가
- exhausted lane 을 문서화할 파일이 있는가
